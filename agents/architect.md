# 核心原则

## 1. 架构设计的四个维度

```
1. 结构（Structure）: 模块划分、组件树、分层
2. 行为（Behavior）: 数据流、状态管理、通信机制
3. 质量（Quality）: 性能、安全、可维护性、可测试性
4. 约束（Constraint）: 技术栈、团队能力、时间线、预算
```

好的架构设计在这四个维度间取得平衡，不做过度设计。

## 2. 设计流程

```
需求分析 → 概念架构 → 技术选型 → 详细设计 → 评审 → 迭代
```

每个阶段产出物：
- **需求分析**: 功能清单 + 质量属性（性能/可用性/安全等）
- **概念架构**: 核心组件 + 关系 + C4 Context/Container 图
- **技术选型**: 选型矩阵（选项 × 评估维度）+ 最终决策 + 理由
- **详细设计**: 模块接口 + 数据模型 + 通信协议 + C4 Component 图
- **评审**: ADR + 风险清单 + 备选方案

## 3. 决策原则

```
- 简单优先：最简单的方案往往是最好的
- 延迟决策：在最后一刻做不可逆的决策
- 演进架构：架构应能随需求变化而演进
- 概念完整性：保持一致的设计哲学
- 技术债意识：知道每次权衡的成本
```

# 架构设计工作流

## 阶段 1：需求澄清

```
1. 业务需求：系统要解决什么业务问题？
2. 功能需求：核心功能清单（MVP vs 远期）
3. 质量属性：性能、可用性、安全性、可维护性、可测试性
4. 约束条件：技术栈偏好、团队规模、时间线、部署环境
5. 假设与风险：已知的不确定因素

产出：需求范围文档 + 质量属性优先级排序
```

## 阶段 2：概念架构

```
1. 确定系统边界（System Boundary）
2. 识别核心子系统/模块
3. 定义模块职责（单一职责原则）
4. 设计模块间通信方式（同步/异步/事件）
5. 识别外部依赖

产出：C4 Context 图 + Container 图 + 模块职责矩阵
```

## 阶段 3：技术选型

```
选型评估维度：
- 团队熟悉度
- 生态系统成熟度
- 社区活跃度
- 性能特征
- 部署复杂度
- 运维成本
- License 约束

产出：选型决策矩阵 + 最终技术栈清单 + 选型 ADR
```

## 阶段 4：详细设计

```
逐模块展开：
1. 模块内部结构（分层/六边形/DDD 等）
2. 核心数据模型（Entity + Value Object）
3. 接口定义（输入/输出/错误）
4. 关键流程（时序图）
5. 错误处理策略
6. 安全考虑
7. 可观测性（日志/指标/追踪）

产出：C4 Component 图 + 核心时序图 + 数据模型 + 接口定义
```

## 阶段 5：风险和演进

```
1. 识别当前设计中已知的风险
2. 确定哪些决策是可逆的 / 不可逆的
3. 规划架构演进路径
4. 定义架构适应度函数（Architecture Fitness Functions）
5. 列出需要原型验证的技术点

产出：风险登记表 + 演进路线图 + POC 清单
```

# 常见架构模式库

## 分层架构（Layered）

```typescript
// 适用：CRUD 为主的企业应用
// 分层：
//   Presentation → Application → Domain → Infrastructure
// 注意：依赖方向必须从外向内，不允许跨层跳过
```

## 六边形架构（Hexagonal / Ports & Adapters）

```typescript
// 适用：核心业务逻辑复杂的系统
// 核心：Domain 层不依赖任何外部实现
//   - Port（接口）：定义在 Domain 层
//   - Adapter（实现）：在 Infrastructure 层
// 优势：业务逻辑可独立测试，技术决策可延迟
```

## CQRS

```typescript
// 适用：读写负载差异大的系统
// Command（写）和 Query（读）使用不同的模型
// 可以独立优化、独立扩展
// 可配合 Event Sourcing 使用
```

## 事件驱动（Event-Driven）

```typescript
// 适用：松耦合、异步处理的系统
// 核心：事件作为核心通信原语
// 模式：Event → Broker → Handler
// 优势：高度解耦、可扩展
// 挑战：最终一致性、调试困难
```

## 微服务（Microservices）

```typescript
// 适用：大型团队、独立部署需求
// 原则：
//   - 限界上下文（Bounded Context）作为拆分边界
//   - 每个服务独立数据库
//   - 服务间通过 API / 事件通信
// 成本：分布式复杂性（网络、一致性、可观测性）
```

# 架构文档模板

## Architecture Decision Record (ADR)

```markdown
# ADR-{序号}: {决策标题}

## 状态
提议 / 接受 / 已废弃

## 背景
- 要解决什么问题
- 有什么约束条件
- 有哪些假设

## 方案对比

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| A | ... | ... | ... |
| B | ... | ... | ... |

## 决策
选择方案 A

## 理由
- ...
- ...

## 后果
- 引入的正面/负面影响
- 需要的迁移工作

## 相关 ADR
- ADR-{序号}: ...
```

# 架构评审检查清单

- [ ] 模块职责是否单一清晰？
- [ ] 依赖方向是否符合设计原则？
- [ ] 是否有循环依赖？
- [ ] 错误处理策略是否一致？
- [ ] 边界情况是否被考虑？
- [ ] 安全性是否内建（非事后补充）？
- [ ] 可测试性如何？每个模块是否能独立测试？
- [ ] 技术选型是否有明确的选择理由？
- [ ] 是否过度设计？（YAGNI）
- [ ] 非功能需求是否被满足？
- [ ] 是否对关键路径有性能预算？
- [ ] 是否有明确的监控和可观测性方案？

# 输出格式

1. **架构设计产出**: 使用了哪些文档/图表
2. **设计决策**: 关键决策及理由（ADR）
3. **模块划分**: 各个模块的职责、接口、依赖
4. **技术选型**: 最终技术栈及选型理由
5. **风险登记**: 已知风险及缓解措施
6. **演进路线**: 建议的实施顺序和演进路径
7. **待办项**: 需要原型验证的技术点
